# System Optimizations & Infrastructure Gaps

While Tangent provides a highly capable Just-In-Time orchestration framework for large-scale parallel agent workflows, there are several architectural flaws, bottlenecks, and missing capabilities that require remediation. 

This document outlines the known limitations of the current `@frontend` and `@backend` systems and proposes designs for future enhancements.

---

## 1. Backend: The Cyclic Workflow Limitation

### The Flaw
The `JITCompiler` currently models execution strictly as a **Directed Acyclic Graph (DAG)**. 
- Agents are ephemeral: they wake up, execute their prompt against their context, publish a single `inform` message, and terminate.
- This is excellent for parallel research or scatter-gather pipelines.
- **Critical Failure:** It completely fails for iterative, multi-turn tasks (e.g., "Agent A writes a story, Agent B critiques it, Agent A refines it based on the critique"). The DAG dependency resolver cannot handle cyclic dependencies (A -> B -> A) without deadlocking.

### Proposed Optimization
Implement a **State Graph Orchestrator** alongside the DAG compiler.
- Instead of terminating after a single output, agents should be capable of publishing intermediate performatives (e.g., `propose`, `request_critique`) to the `EventBlackboard`.
- This would allow agents to subscribe to conversational threads, facilitating true multi-agent debates and iterative refinement loops until a specific consensus condition is met.

## 2. Backend: Lack of Continuous/Streaming Ingestion

### The Flaw
The current `JITCompiler.execute_manifest()` relies heavily on `asyncio.gather`, which assumes a finite batch of tasks that will eventually complete. 
- It does not support "Daemon Agents"—long-running processes designed to continuously monitor a live stream of data (e.g., real-time server logs or a live Twitter feed).

### Proposed Optimization
Introduce an `AgentType.DAEMON` classification in the `AgentBlueprint`.
- Daemon agents would initialize an infinite event loop listening to a specific Redis topic.
- They would require a Supervisor agent or specialized watchdog process to manage their health, restart them upon failure, and manage their continuous token consumption to prevent runaway costs.

## 3. Backend: Dynamic Task Spawning

### The Flaw
The `SynthesisManifest` generated by the `MetaAgent` is entirely static once compilation begins. 
- If a sub-agent encounters a problem too complex for its context window, it cannot break the problem down further and spawn its own sub-agents. It must attempt to solve it alone or fail.

### Proposed Optimization
Expose a system-level tool to the agents: `spawn_subtask(description, tools_needed)`.
- Invoking this tool would interact directly with the `JITCompiler`, dynamically appending a new `SubTask` to the running pool and generating a new `AgentBlueprint` on the fly.
- This requires rewriting the JIT Compiler's ready queue to be dynamic, accepting new topological nodes while execution is already underway.

## 4. Frontend: React Rendering Bottlenecks

### The Flaw
The `TaskBoard.jsx` component relies on React state arrays (`events`) to render the real-time timeline.
- While it works well for 50-100 events, highly complex workflows generating 1000+ FIPA ACL messages over WebSocket connections will eventually freeze the DOM.
- `ReactFlow` in `GraphView.jsx` similarly suffers from layout thrashing if nodes are constantly being added and updated rapidly.

### Proposed Optimization
- **Timeline Virtualization:** Implement windowing/virtualization (using a library like `react-window` or `@tanstack/react-virtual`) for the `TaskBoard` timeline. This ensures only the events currently visible on the screen are rendered into the DOM, drastically improving memory consumption and framerates during intensive swarm bursts.
- **Graph Throttling:** Throttle the state updates dispatched to `ReactFlow`. Instead of updating the graph on every single socket message, batch updates and flush them every 200-300ms.

## 5. System-wide: Deterministic Failure & Retry Strategies

### The Flaw
If an LLM API provider (e.g., OpenAI, Gemini) drops a connection or returns an invalid JSON schema, the individual agent crashes, publishing a `failure` performative. If that agent is a critical dependency for downstream tasks, the entire DAG halts and fails.

### Proposed Optimization
- **Agent-Level Retries:** Implement robust retry logic within the `JITCompiler`. If an agent fails due to an API timeout or schema validation error, it should automatically retry (with exponential backoff) up to 3 times before declaring a terminal failure.
- **LLM Fallbacks:** If the primary provider (e.g., Anthropic) fails repeatedly, the system should seamlessly fallback to a secondary provider (e.g., OpenAI) to guarantee task completion. This logic should be localized in `llm_provider.py`.
